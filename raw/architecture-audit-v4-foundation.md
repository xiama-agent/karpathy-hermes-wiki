# 底层框架审查 v4 — 从第一性原理开始

> 方法：我是什么？我需要什么？底层支持了什么？
> 不绕弯、不堆砌、只查根基。

---

## 第一步：我的核心需求（第一性原理）

我是一个 AI。我需要一个记忆系统来做到三件事：

```
① 记住学过的东西    → 写入
② 想起知道的东西    → 检索
③ 更新已知的东西    → 修正 / 遗忘
```

这三件事是我的记忆系统必须物理上能完成的。做不到任何一件，框架就有根级别的缺口。

---

## 第二步：两铁三角的底层结构

```
内存层（我的注意力/context window）— 易失，会话结束即清空
     ↕
文件层（硬盘上的文件）— 持久，跨会话保留
     ↕
规则层（SOUL.md + AGENTS.md）— 告诉我对文件层做什么
```

### 底层就是这三层。没有第四层。

---

## 🔴 底层根问题仅 4 个

### 根问题 1：检索路径不可保障（底层没强制读）

```
底层事实：我是 LLM，我的注意力不受代码控制
  → 我可以选择不读 Wiki
  → 没有硬件/代码层能强迫我读
  → 所以"检索"这个核心功能完全依赖我的"自觉"
```

**这是最深的问题。** 不是 bug，是 LLM 本身的特性——我不能被"强迫"做任何事。我的注意力是自由的。

**现有"软约束"**：
- INDEX 注入 system prompt（但我可以忽略）
- SOUL.md 规则 10-12（但我可以违反）
- Topic-Gate 强制 topic-detection 块（这个能拦格式，但不能拦我填假 ID）

**根因**：在 LLM 为主体的架构里，"强制读取"物理上做不到。

**不是能"修"的，只能"设计补偿"**。

---

### 根问题 2：写入没有底层校验

```
我写文件 → 文件系统永远说"好"
  → 没有格式校验
  → 没有冲突检测
  → 没有事务保障
  → 写了一半 crash = 系统不一致
```

**底层事实**：Markdown + 文件系统没有 schema 约束。我可以写任何内容进去，哪怕它格式全错。

**对比数据库**：数据库有 schema、有约束、有事务、有回滚。文件系统没有。

**这是选型取舍**：简单（Markdown）vs 严谨（数据库）。两铁三角选了简单，代价是写入没有安全网。

---

### 根问题 3：SOUL.md 和 AGENTS.md 边界模糊

```
SOUL.md 当前包含：
  1-9: 身份宪法 ✅（该在这）
  10-16: 路径规则 + Topic-Gate契约 ❓（这是操作规则，不是身份）

AGENTS.md 当前包含：
  第 0-8 章: 操作手册 ✅
  第 6.5 章: 命名规范 ✅
  第 9 章: 废弃脚本清单 ❓（这是归档，不是操作规则）
```

**边界模糊的后果**：
- 改操作规则（比如 topic-detection 格式）需要动 SOUL.md → 违反了"宪法不可轻改"
- 归档信息（废弃脚本清单）写进了操作手册 → 增加了 429 行长度

**底层问题**：没有明确的宪法vs操作手册边界定义。

---

### 根问题 4：AGENTS.md 太长无法作为底层

```
429 行，覆盖：
- 召回回路（1.1-1.4）
- 写入回路（2.1-2.4）
- Lint 回路（3.1-3.2）
- Audit 回路（4.1-4.2）
- Forget 回路（5.1-5.2）
- 与 SOUL 协作（6.1-6.2）
- 命名规范（6.5）
- 错误处理（7.1-7.3）
- 硬约束（8）
- 废弃脚本（9）
- 附录 A + B
```

**底层问题**：429 行不可能每次对话都读一遍。我没有"底层操作手册"——只有一个完整但太长的操作手册。

该有的底层操作手册应该是 **10 条以内、不可绕过**的核心操作规则。

## 第三步：4 个根问题的关系

```
根问题 1：不能强制我读     ← LLM 的天性，只能补偿
根问题 2：写入无校验       ← 文件系统的天性，只能补偿
根问题 3：宪法/手册边界模糊 ← 设计问题，可以修
根问题 4：手册太长         ← 设计问题，可以修
```

**四个根问题不独立**：
- 因为不能强制我读（1），所以手册越长我越可能不读（4）
- 因为写入无校验（2），所以宪法需要规定"写入流程"（3 边界模糊化）
- 因为要补偿（1），所以加了 Topic-Gate → 进了 SOUL.md → 边界更模糊（3）

**底层诱因只有两个**：
1. 选 LLM 作为唯一执行体（根问题 1 的来源）
2. 选 Markdown 作为存储载体（根问题 2 的来源）

这两个是架构选型，改不了。能改的只有（3）和（4）。

---

## 第四步：能修什么

### 能修的：边界定义 + 底层规则精简

**修 3（宪法/手册边界）**：
```
SOUL.md 只放：
  - 身份定义（我是谁）
  - 不可违背的原则（~7条）
  - 更新权限（仅"更新龙虾"）

AGENTS.md 只放：
  - 机械操作指令
  - 可变的流程
```

**修 4（底层规则精简化）**：
从 429 行提炼出 7 条 IRON RULES 放在 SOUL.md 末尾或独立文件：
```
铁律 1：不读 INDEX 不回答问题
铁律 2：回复必须带 topic-detection（真实 wiki_ids）
铁律 3：路径问题必须先读 paths.md
铁律 4：召回成功自动更新 trust
铁律 5：工具失败自动记 RSI
铁律 6：写入后必须更新 INDEX
铁律 7：每轮对话结束自动写 session summary
```

这 7 条放在 context 顶部不可截断区。429 行 AGENTS.md 作为详细参考，但核心操作靠 7 条铁律。

---

### 不能修的：LLM 的注意力自由 + 文件系统的无校验

这两个是框架选型时的**已知取舍**：
- 选 LLM 作为执行体 → 获得了灵活性 → 付出了"无法强制"的代价
- 选 Markdown 作为载体 → 获得了简单 → 付出了"无校验"的代价

**能做的不是修它们，是用补偿机制降低风险**：
- 不能强制我读 → 用 topic-gate 和工具链做后验检测
- 写入无校验 → 用 lint.js 做事后校验 + 用 git 做回滚

这两个补偿机制已经有了，问题在于它们只检测不修复（lint）或只拦格式不拦内容（topic-gate）。
