# 底层框架审计报告 — 两铁三角 v3.1.5

> 日期：2026-06-26
> 范围：架构设计缺陷（非 bug，非操作 gap）
> 方法：逐层审查 SOUL.md → AGENTS.md → Wiki → 工具链 → 执行路径

---

## 第一层：两铁三角的"第二个三角形"在哪？

**两铁三角**这个名字暗示有两个三角形。目前只有一个：
```
SOUL.md → AGENTS.md → Wiki    ← 这是"知识三角形"
                                 ← 第二个三角形不存在
```

旧四层系统其实有双三角：
```
三角形1：SOUL → memory → fact_store（知识通道）
三角形2：skills → Harness → 执行引擎（执行通道）
```

重构后第二个三角形消失了。`skills/` 变成了 22 个孤立的 SKILL.md，没有统一执行引擎。Harness 被标记为 deprecated。`tools/` 目录不存在。

**结论**：两铁三角实际是 **"一铁三角 + 一堆技能"**。第二个铁三角（执行/反馈通道）从未建立。

## 第二层：4 个致命的架构设计缺陷

### 🔴 II-1. 单点执行故障（LLM 独裁）

```
所有操作走我 → 我 skip → 系统归零
     ↑
没有副驾驶、没有第二执行路径、没有后备检索
```

旧系统的 fact_store（SQLite）是**独立于 LLM 的检索层**。我状态不好时它还能工作。
新系统的 Wiki（Markdown）依赖 **LLM 注意力**才能被读取。

如果 DeepSeek 今天 context 满了、token 窗口挤了、或者我犯懒——没有第二通道可以访问知识。

**严重度**：致命。这是选型问题，不是实现问题。

### 🔴 II-2. 动态机制放进了静态载体（信任环物理上不通）

```
需求：trust += 0.1 每次召回成功    ← 动态操作
载体：Markdown 文件里的一个数字     ← 静态存储
成本：read_file → 找到429行中的1行 → patch → git commit
对比DB：UPDATE facts SET trust = trust + 0.1 WHERE id = 'X'
```

AGENTS.md 1.4 写了规则，`update-trust.js` 写了工具。但 04-facts/ 是 1652 行的纯文本——每次信任更新需要：
1. 定位正确文件（4 个中选 1）
2. 定位正确行（988 行中找 1 行）
3. 精确 patch
4. git commit

旧 fact_store（SQLite）是一条 SQL 的事。**新架构选择了更差的工具来做这个工作**。

信任环 18 项修复后仍然不通——不是没修好，是设计上就不该让 Markdown 干数据库的活。

### 🔴 II-3. 写入没有校验层（我是自己审自己）

```
我决定写 → write_file → 写入完成
                  ↑
           没有人检查格式、完整性、冲突
```

- frontmatter 格式？→ lint 事后 6h 才知道
- INDEX 更新？→ 靠我记得
- wikilink 通不通？→ 靠我记得
- 跟已有内容矛盾？→ 没有人实时检查
- 写入过程中断（写了一半 crash）？→ 没有事务保障

旧系统有 `link_facts_to_wiki.py` 做写入后校验。现在 `.disabled` 了，无替代。

### 🔴 II-4. 单脑冒充多脑（AI 调度是愿望不是现实）

SOUL.md 写了完整的 AI 调度系统：
- 龙虾是管家
- Mavis 是执行手
- Codex 未来加入
- discovery 共享池
- 任务单写到 `tasks/mavis/`

**实际运行**：
| 组件 | 状态 |
|:----|:----:|
| 我读 Wiki | ✅ |
| Mavis 写 Wiki | ✅（今天刚确认） |
| Mavis **读** Wiki | ❓（.mavis_read_state.json 存在但不确认读了多少） |
| discovery 共享池 | ❌ 4 个目录，无人真正"每次会话开始扫描" |
| 任务单双向 | ❌ 单向（我→Mavis），没有 Mavis→我的结果回写机制 |
| Codex | ❌ 不存在 |

**这是一个多体系统架构，跑在单体运行时上。**

## 第三层：9 个框架设计纰漏

### 🟡 III-1. 无会话间状态传递

每天第一次对话的我，不知道昨天发生了什么。除非：
- YANG 告诉我
- 我去读 git log
- 有人手动写了日报

没有**会话结束仪式**：auto-write session summary → `log.md` → 状态快照。

### 🟡 III-2. 04-facts 文件过大（单项 32KB）

```
general.md: 988 行 / ~32KB
tool.md:    416 行 / ~16KB
```

花 3 秒读完 frontmatter 才知道这页讲什么。要找到一个特定事实需要读完半页。

**架构问题**：单文件聚合 64 条事实 → 每次读取全量加载 → 浪费大量 context。

### 🟡 III-3. Git 是单点故障

- 所有版本历史在 git
- 所有提交者是 `Hermes Brain`（无法区分谁改了啥）
- 没有 per-file version independent of git
- git 仓库损坏 = 所有历史丢失
- 备份是 git bundle，但**从未验证恢复流程是否可用**

### 🟡 III-4. 无热/温/冷数据分层

| 层 | 内容 | 频率 | 现状 |
|:---|:-----|:----:|:----|
| 🔥 Hot | SOUL.md + INDEX | 每轮 | ✅ 自动注入 |
| 🌤️ Warm | AGENTS.md + 常用页面 | 每次对话 | ❌ 没有缓存机制 |
| 🧊 Cold | 04-facts + 旧存档 | 偶尔 | ❌ 跟 hot 数据同一访问路径 |

所有页面通过同一通道（我的注意力 + read_file）访问。没有缓存、没有预取。

### 🟡 III-5. 无冷启动流程

如果 Wiki 为空（全新安装）：
- INDEX.md 不存在 → crash（"系统未初始化"）
- 没有任何初始数据
- 没有任何引导流程
- 完全依赖 YANG 手动初始化

### 🟡 III-6. 无数据归属与冲突解决

- 同一事实可能同时存在于 02-knowledge/ 和 04-facts/
- 矛盾时：lint 报告 × 等待 YANG 处理
- 没有"数据主人"概念（谁写的？谁认证的？）
- 没有版本优先级（新版本覆盖旧版本的规则）

### 🟡 III-7. 无工具失败 → 记忆反馈

工具调用失败 → 应该写 RSI。但：
- 工具失败 → 我重试 → 成功/放弃
- 没有任何自动触发器把失败记录到 RSI
- RSI 页面有了、格式有了、但数据流是断的

### 🟡 III-8. 无迁移路径

v3.1.5 → v4.0 怎么迁？
- 没有 schema 版本号
- 没有迁移脚本
- 全靠手动重写
- 如果老文件格式变了，新系统不认识旧数据

### 🟡 III-9. 定时任务状态不明

3 个 Windows 定时任务（Lint/Log Init/Backup）：
- `schtasks /query` 返回空 → 不确定是否注册成功
- 没有 keepalive 检查
- 没有失败通知
- 没有看门狗

## 第四层：12 个之前漏掉的 bug/缺陷

### 🔵 IV-1. AGENTS.md 429 行太长了

每次要理解操作手册需要读 429 行。没有 TL;DR、没有分片读取机制。

### 🔵 IV-2. 99-temp 混装

candidates / lint 报告 / missed-recall / audit 副本全在一个目录。文件名命名有三种格式（candidates- / candidate- / lint-），没有统一规范。

### 🔵 IV-3. discovery/ 无人维护

4 个目录，最近更新 2026-06-26 16:06（workflows），但没有任何 AI 真正"每次会话开始扫描 discovery/"。
`README.md` 写了规则但无人执行。

### 🔵 IV-4. lint.js 不 lint 自己

lint 检查 Wiki 健康，但不检查自己的代码有没有过时。

### 🔵 IV-5. update-trust.js 无回滚

改了 trust 没有 undo 命令。改错了只能手动 edit。

### 🔵 IV-6. 无性能指标

recall.js 跑一次要多久？lint.js 要多久？不知道。没有 timing。

### 🔵 IV-7. 无告警

lint 发现问题 → 写入文件。没有推送给 YANG 的机制。如果不是 YANG 主动看，坏了也不知道。

### 🔵 IV-8. "Hermes Brain" 是唯一的 git 作者

无法区分是我的修改还是 Mavis 的修改还是 YANG 的手动修改。

### 🔵 IV-9. 写入无事务

写入 Wiki 不是原子操作。如果写了一半中断（git commit 前断电），系统进入不一致状态。

### 🔵 IV-10. 无写入频率控制

如果一小时内 write_file 20 次，会产生 20 次 git 操作和 20 个 commit。没有 commit 合并机制。

### 🔵 IV-11. 04-facts 的 `source` 字段模糊

`source: fact_store#23` 或 `source: SOUL.md L7`。指向旧系统位置但不指向当前 Wiki 页面。溯源链断了。

### 🔵 IV-12. 无容量规划

Wiki 现在 250KB。能长到多大？500KB？2MB？10MB？什么时候需要拆分？没有阈值。
