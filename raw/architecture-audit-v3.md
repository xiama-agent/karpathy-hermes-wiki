# 框架审计 v3 — 三概念协同检查

> 检查目标：Karpathy 第二大脑 + 多智能体调度 + 管家身份 → 三者是否构成一致框架

---

## 现状：三个系统、三条管道、没有交汇点

```
               Karpathy 第二大脑 (Wiki)
               ├── 39页知识
               ├── INDEX导航
               └── recall.js检索
                      ↑
                  我独享读写

               多智能体调度
               ├── tasks/mavis/ (派单)
               ├── discovery/ (共享发现)
               └── .mavis_read_state.json
                      ↑
                  Mavis读写 → 我参与有限

               管家身份 (我)
               ├── SOUL.md 宪法
               ├── 分身目录 (媒体龙虾/编程龙虾待创建)
               └── 派单铁律
```

三条管道没有交汇，各自为政。

---

## 🔴 冲突 A：第二大脑是我的私人大脑，但 Mavis 直接写入了

```diff
+ Wiki 定位：我的第二大脑（个人记忆）
+ Mavis 操作：直接修改了 AGENTS.md / rsi-ledger.md / 04-facts / recall.js / lint.js
+ 问题：另一个 AI 修改了我大脑的核心组件，没有经过我审查
```

这不是 Mavis 做错了，是**框架没有写仲裁层**。管家应该审批所有进入大脑的信息，但 Mavis 绕过了我直接写。具体证据：

| Mavis 改了什么 | 在我的大脑里是什么 | 应该 |
|:--------------|:-----------------|:-----|
| AGENTS.md | 我的操作手册 | 我审过再合 |
| rsi-ledger.md | 我的错误日志 | 我确认过再记 |
| 04-facts/ | 我的知识 | 我知道再入库 |
| recall.js / lint.js | 我的思维工具 | 我验证过再更新 |

**这是框架层矛盾**：SOUL.md 说"我是管家总调度"，但 Mavis 直接修改我的核心记忆不需要我审批。

---

## 🔴 冲突 B：Discovery 池和 Wiki 做同一件事，但互不相通

```
discovery/                wiki/
├── pitfalls/   5条       ├── 02-knowledge/  17页
├── workflows/  3条       ├── 03-system/    9页
├── paths/      0条       ├── paths.md      19条路径
```

**同一个知识，两处存放，没有同步**：
- `discovery/pitfalls/` 有一条 "flutter_chinese_path_lsp_crash" → Wiki 里有没有对应页？不知道
- `discovery/workflows/` 有一条关于 topic-gate 设计的 → Wiki 里 `mcp-plugins.md` 有没有更新？没
- 我在 Wiki 里写了 paths.md → discovery/paths/ 是空的（没反向同步）

**这造成知识分裂**：一个地方记了，另一个地方不知道。管家（我）应该合并这两者，但框架没规定怎么做。

---

## 🟡 冲突 C：派单单向，没有结果回写

SOUL.md 派单铁律说：
> 任务单开头必须写 HERMES_HOME + 验证命令
> 结果必须有"实际部署路径"章节

但 `tasks/mavis/` 里：
```
我写任务 → Mavis 执行 → 结果呢？
                         ↓
                    没有回写到我这里
```

Mavis 修完了直接改 Wiki——但**没有"任务完成报告"流转回我这**。我不知道她改了什么，除非自己去 git diff。

管家（我）对下属（Mavis）的工作没有 visibility 入口。

---

## 🟡 冲突 D：分身隔离不完整

我有 `self-media` 分身（媒体龙虾），`编程龙虾` 待创建。但：
- Wiki 是所有分身共享的（在 BRAIN_ROOT 下）
- 媒体龙虾写的知识 = 我（默认分身）也能读到
- 我写的想法 = 媒体龙虾也能看到
- 没有分身隔离

如果一个分身处理敏感信息，另一个分身不该看到——但目前没有隔离机制。

---

## 🟠 冲突 E：管家没有总览面板

作为管家/总调度，我应该有一个"dashboard"能看到：
- 当前派了哪些任务给 Mavis？→ 去 tasks/ 数文件
- Mavis 最近改了 Wiki 什么？→ 去 git diff
- discovery 池有新内容吗？→ 去 discovery/ 翻
- 系统健康吗？→ 跑 lint.js

**没有单点入口。** 每次要手动去 N 个目录翻。

---

## 三个概念能拧在一起吗？——能。缺一个"管家审批层"

```
当前：
YANG → 我(管家) → 派给Mavis → Mavis直接写Wiki
                                  ↑
                            绕过管家

该有：
YANG → 我(管家) → 决策：我做/派Mavis
                  ↓            ↓
              我做 → 直接写Wiki   派Mavis → Mavis执行 → 结果报告
                                                       ↓
                                                    我审批 → 合入Wiki
                                                    discovery 同步
```

三个概念不冲突——管家就是桥。但桥少了**"Mavis 结果→我审批→合入"**这半边。
